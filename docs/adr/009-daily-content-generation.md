# ADR 009 ŌĆö Daily content generation: schedule, isolation, ranking, telemetry

## Status
Accepted

## Context
Phase 9 introduces a Lambda (`diamond-iq-generate-daily-content`) that
produces three categories of editorial copy from yesterday's Final
games and today's Preview games:

- **Recap** ŌĆö one short editorial per Final game.
- **Preview** ŌĆö one anticipatory blurb per Preview game.
- **Featured** ŌĆö extended analysis on the day's two most-interesting
  matchups, ranked by a heuristic.

Each item is generated by a single Bedrock InvokeModel call against
Claude Sonnet 4.6 (see ADR 008) and written to DynamoDB under
`PK=CONTENT#<date>` with one of three SK shapes
(`RECAP#<game_pk>` / `PREVIEW#<game_pk>` / `FEATURED#<rank>`). The
API Lambda reads these via `GET /content/today` and the React frontend
renders them on the home page.

Six design questions had to be settled before this Lambda could ship:

1. How often it should run.
2. What happens when one Bedrock call fails inside a multi-call run.
3. How to pick which two games are "Featured."
4. How to make repeated runs safe (idempotency).
5. What custom telemetry to emit.
6. Anthropic via Bedrock vs Anthropic API directly.

## Decision

### 1. Three EventBridge ticks at 15:00 / 16:00 / 17:00 UTC

The Lambda runs three times per day instead of once:

```
cron(0 15 * * ? *)   # primary
cron(0 16 * * ? *)   # retry-1
cron(0 17 * * ? *)   # retry-2
```

The 15:00 UTC tick is "primary"; the 16:00 and 17:00 ticks are
opportunistic fillers. Combined with the idempotency check (decision
4), the second and third invocations are no-ops on a healthy run, but
recover automatically when the first invocation hit a transient
Bedrock error and only generated some of the items.

This converts a class of partial failure from "operator must manually
re-invoke" to "the system fixes itself within two hours." Real-world
case: throttling that briefly affects one or two of fifteen invocations.

### 2. Item-level error isolation

Inside a single invocation we may call Bedrock 15+ times (one per
Final + Preview game + 2 featured). A single ThrottlingException on
the third call must not abort the seventh. The handler runs each
content item through its own try/except chain:

```python
for game in yesterday_finals:
    _generate_one(...)  # catches ClientError, generic Exception,
                        # increments bedrock_failures, continues
```

Three counters in the summary:

- `items_written` ŌĆö DynamoDB writes that succeeded
- `bedrock_failures` ŌĆö InvokeModel errors (logged at WARNING for
  transient codes, ERROR otherwise)
- `dynamodb_failures` ŌĆö write errors after a successful Bedrock call

The summary's `ok` field is `True` if anything was written OR if
nothing failed. `False` only when the run was a total failure.

### 3. Featured heuristic

Two slots; ranked from today's Preview slate by:

```
score(game) =
  +2.0  if away.division == home.division          # rivalry weight
  +1.0  per team that won its game yesterday        # form weight
  +0.5  if home team plays in Pacific Time          # marquee/audience weight
```

Tie-break: smaller `game_pk` wins (deterministic, stable across runs).
Top 2 selected. PT teams are LAA, OAK, SEA, LAD, SD, SF ŌĆö the six
clubs in the actual Pacific time zone, not the broader "AL/NL West"
geographic divisions (HOU and TEX are Central Time and don't get the
late-night-national-broadcast premium).

### 4. Idempotency

Before the per-item loop, the Lambda queries DynamoDB:

```python
existing_sks = list_existing_content_sks(today)  # set[str], one query
expected_sks = {RECAP#<pk>...} | {PREVIEW#<pk>...} | {FEATURED#1, FEATURED#2}
missing_sks  = expected_sks - existing_sks
```

The per-game loops skip any SK already in `existing_sks`, so Bedrock
is only called for items that don't yet exist. A successful first run
followed by a successful second run results in **zero** Bedrock
invocations on the second run; a half-successful first run followed
by a second run completes only the missing pieces.

### 5. Custom telemetry

Four metrics, namespace `DiamondIQ/Content`, dimension
`LambdaFunction = diamond-iq-generate-daily-content`:

- `BedrockFailures` (Count)
- `DynamoDBFailures` (Count)
- `ItemsWritten` (Count)
- `ItemsSkipped` (Count)

Emitted as one batch `put_metric_data` call at the end of every
invocation (including the early-exit and idempotent-skip paths).
Wrapped in `try/except` so a CloudWatch outage never fails the
Lambda.

These four enable two CloudWatch alarms (`bedrock-failures > 0 / 1h`
and `dynamodb-failures > 0 / 1h`) that would not be possible from
the default `AWS/Lambda` namespace alone, since the handler catches
those exceptions per-item and never lets them surface as
`AWS/Lambda#Errors`.

### 6. Bedrock (not Anthropic API direct)

The Lambda invokes Claude through Amazon Bedrock's
`bedrock-runtime` API rather than the Anthropic API at
`api.anthropic.com`.

## Consequences

### Positive
- **Self-healing schedule.** Operators don't manually re-invoke
  failed runs; the next scheduled tick finishes the job.
- **One bad item doesn't kill the run.** The dashboard sees 14 of
  15 recaps even on a partially-throttled afternoon.
- **Featured ranking is explainable.** A reviewer can read the
  heuristic, point at the chosen games, and verify the math by hand.
  This matters more for a portfolio project than ML-grade ranking
  quality would.
- **Custom metrics give us alarms the default Lambda metrics can't.**
  Per-item Bedrock failures are visible to CloudWatch even though
  they never raise `AWS/Lambda#Errors`.
- **Bedrock unifies cloud bill and IAM auth.** No separate Anthropic
  account or API key to manage; the Lambda's IAM role is the
  identity, no secrets stored anywhere.
- **Resume signal.** A cloud-engineering portfolio benefits more
  from "I shipped Bedrock + IAM + cross-region inference profiles"
  than from "I called an HTTPS API."

### Negative
- **Bedrock has account-level quota gates.** A new AWS account must
  submit Anthropic's use-case form (ADR 008), and per-account daily
  token quotas can be locked at zero until manually reviewed by AWS
  Support. The Anthropic API direct does not have this gate.
- **Three EventBridge invocations per day is 3x the steady-state
  Lambda cost.** At our volume the difference is fractions of a
  cent; not material but non-zero.
- **Custom CloudWatch metrics cost $0.30 / metric / month** (after
  free tier). Four metrics ├Ś one Lambda = $1.20/month worst case.
  Acceptable for the operational visibility they unlock.
- **Idempotency check costs one DynamoDB query per run.** Three
  queries per day ├Ś 30 days Ōēł 90 queries/month. Trivial.

### Operational notes
- A failed first tick that emits structured `bedrock_failures > 0`
  triggers the `bedrock-failures` CloudWatch alarm within a 1-hour
  evaluation window. The 16:00 and 17:00 ticks may complete the
  missing items before the alarm even pages anyone, in which case
  the alarm self-resolves to OK and the operator gets a paired
  ALARM/OK email pair. That's working as intended.
- If the heuristic ever produces a Featured choice that looks
  wrong, the fix is a one-line tuning of the score weights, not a
  retraining cycle.

## Alternatives considered

### Schedule

- **Once-daily at 15:00 UTC.** Simpler, but a single Bedrock blip
  during that one window means a missing recap until the next day.
  Rejected.
- **Hourly all day.** Pure overkill; content updates daily. Each
  extra tick after the first is just a no-op idempotent query.
  Rejected.
- **EventBridge `rate(N hours)` instead of three explicit `cron`
  rules.** Less precise scheduling (drifts based on when terraform
  applied first); the explicit cron triple is more diagnosable.
  Rejected.

### Error isolation model

- **Fail-fast: first error aborts the run.** Simpler code, but loses
  all in-progress work. Rejected for the partial-success case.
- **Re-raise per item, let Lambda retry the whole invocation.** AWS
  Lambda's retry-on-error semantics for asynchronous invocations
  would re-run the entire batch including already-written items.
  Idempotent writes mean this isn't catastrophic, but it triples
  Bedrock cost on a single bad item. Rejected.

### Featured ranking

- **ML model trained on engagement data.** No engagement data exists
  yet (portfolio project, no live audience). Rejected.
- **Random selection.** Trivially explainable but produces poor
  Featured choices when the slate has a clear marquee game.
  Rejected.
- **Operator-curated.** Defeats the "AI does it daily" premise.
  Rejected.
- **Larger heuristic with more factors (pitcher matchup, head-to-head
  history, standings impact, weather).** Each new factor needs
  reliable data we don't currently ingest. Deferred until ingestion
  is broadened; the current 3-factor heuristic is honest about what
  it knows.

### Idempotency model

- **Conditional writes (`PutItem` with `attribute_not_exists(SK)`).**
  Wastes one Bedrock call per existing item. Rejected.
- **Per-run UUID in the SK.** Doesn't dedup at all; each run writes
  a new copy. Rejected ŌĆö content table would balloon.
- **No idempotency, just trust the schedule.** Three runs per day
  means 3x cost; throttle blips in either direction (first OK,
  second mid-throttle) lose work. Rejected.

### Telemetry surface

- **Just `AWS/Lambda#Errors`.** Misses the per-item Bedrock failure
  pattern that the handler catches. Rejected.
- **Custom CloudWatch logs filtering with metric filters.** Doable
  but fragile ŌĆö requires log-format coupling between handler and
  metric filter, and metric filters cost the same as `PutMetricData`.
  Rejected.
- **External APM (Datadog, New Relic).** Too heavy for a portfolio
  project; an extra vendor relationship for what's effectively four
  counters. Rejected.

### Bedrock vs Anthropic API direct

- **Anthropic API at `api.anthropic.com` directly.** Pros: one
  fewer hop; no AWS account-level quota gate; available outside
  AWS regions. Cons: separate API key (need a secret store +
  rotation); separate billing relationship; less of a "cloud"
  resume signal. The deciding factor was the resume signal ŌĆö
  this is a cloud-engineering portfolio. Rejected on those grounds.
- **OpenAI / Gemini.** Off-topic for "Anthropic on AWS" framing of
  this project. Not seriously considered.
